# CLAUDE.md

## 项目目标概述

THA 年度赛事积分系统旨在围绕网球赛事积分、排行榜、赛事管理、选手管理与相关运营流程，建立一个可研究、可设计、可实现、可部署的系统化平台。

当前阶段仅进行项目初始化与规范搭建，不进入业务实现。

## 项目目录职责边界

### `frontend/`

- 承载所有前端实现
- 负责用户端与管理端页面、交互、状态管理、API 调用和前端构建
- 不负责业务文档编写
- 不得伪造后端真实能力

### `backend/`

- 承载所有后端实现
- 负责 API、业务逻辑、数据访问、鉴权、集成与后端测试
- 不负责业务文档编写
- 不得擅自改变 API 契约

### `docs/`

- 承载研究、设计、规范、计划与交付文档
- 是所有实现的上游依据
- 每个阶段必须把输出物放到对应的 `docs/` 子目录

## 执行原则

### 复杂任务先计划再执行

当任务跨多个模块、涉及架构或契约变化、或存在明显歧义时，必须先输出计划，列出待确认项，再等待确认后执行。

### 先文档后代码

所有实现必须基于 `docs/` 中已确认的上游文档。未形成文档结论之前，不得提前固化业务代码。

### API 契约优先

后端实现必须以 `docs/08_api_spec/` 为准；如果契约需要变更，必须先更新文档，再修改实现。前端必须按照已确认契约联调，不得自行假定接口结构。

### 前端不得伪造后端能力

前端只能做占位、交互和联调验证，不得用假数据冒充正式能力，不得以本地逻辑替代后端真实业务结果。

### 后端不得擅自改变 API 契约

后端不得在未更新文档的情况下增删字段、修改语义、变更错误码或改变响应结构。

### 主链路优先，MVP 优先

第一阶段优先打通最小可用闭环，先完成主链路，再做非核心增强。

- `docs/05_prd/`：PRD

- `docs/07_data_model/`：数据模型

## 工作方式

- 优先阅读上游文档，再决定实现方式

- 有歧义时优先列出待确认项
- 不写与当前阶段无关的业务代码
- 不越过文档边界自行扩展功能
