# THA 年度赛事积分系统架构设计

## 1. 架构设计目标

本系统面向清华校友网球协会 THA 年度赛事积分体系，核心目标是支撑 2026-2027 年度巡回赛、校庆总决赛、分站赛、卫星赛、团体赛、代表队赛事、办赛奖励和捐赠赞助奖励等多种积分来源的统一管理。

系统不采用 52 周滚动积分，也不采用 Best-of-N 机制，而是以年度赛季为统计周期，将当前赛季内所有合规积分全部计入个人年度总积分。

系统架构需要满足以下目标：

- 支持微信小程序、H5 和 Web 管理后台。
- 用户端定位为公共积分榜查看平台，无需登录，进入即可查看排行榜。
- 支持管理员上传赛事结果 Excel，而不是逐场比分结果。
- 支持单项双打、团体赛、远程参赛奖补、代表队参赛、办赛奖励和捐赠赞助等多种积分来源。
- 支持年度排行榜和个人积分明细展示。
- 支持总积分相同并列排名，同分展示按姓名首字母排序。
- 支持后续接入 AI，用于非标准赛事结果 Excel 的智能解析。
- 保证积分计算可解释、可复现、可审计。

## 2. 推荐技术选型

```text
前端用户端：Uni-app / Vue 3 / TypeScript / Pinia / Wot Design Uni
Web 管理后台：Vue 3 / TypeScript / Element Plus
后端服务：FastAPI / Python / SQLAlchemy 2.0 / Pydantic
数据库：PostgreSQL
缓存与任务队列：Redis + ARQ，或 Redis + Celery
文件存储：本地存储 / 腾讯云 COS / 阿里云 OSS
AI 模块：V3 阶段接入 Dify 或 LangGraph，用于赛事结果 Excel 智能解析
部署方式：Docker + Nginx + 云服务器
```

## 4. 前端架构

## 4.1 用户端

用户端推荐使用：

```text
Uni-app + Vue 3 + TypeScript
```

用户端主要面向普通校友、参赛选手和观众，建议优先支持微信小程序，同时兼容 H5。用户端不做登录体系，也不做“我的页面”，定位为公共积分榜查看平台。

### 4.1.1 用户端核心页面

- 年度积分排行榜首页
- 赛事计划页
- 赛事结果页
- 积分规则说明页

其中，选手个人积分明细不单独设计为“我的页面”或登录后页面，而是在排行榜中点击选手名称后以弹窗、抽屉或半屏面板方式展示。

### 4.1.2 用户端核心能力

- 无需登录，进入即可查看年度总积分排行榜。
- 支持按赛季、组别、院系、姓名筛选排行榜。
- 点击选手名称查看该选手个人积分构成。
- 查看单项赛、团体赛、远程参赛、办赛、捐赠等积分来源。
- 查看校庆总决赛报名和种子排序参考。
- 分享排行榜页面。

### 4.1.3 状态管理

推荐使用：

```text
Pinia
```

用于缓存：

- 当前赛季信息
- 排行榜筛选条件
- 选手基础信息
- 当前选中的选手积分明细
- 弹窗 / 抽屉展示状态

### 4.1.4 UI 组件库

推荐：

```text
Wot Design Uni
```

备选：

```text
uView Plus
Vant Weapp
```

## 4.2 Web 管理后台

管理后台推荐使用：

```text
Vue 3 + TypeScript + Element Plus
```

管理后台主要面向赛事组织者和系统管理员。

### 4.2.1 管理后台核心页面

- 管理员登录
- 赛季管理
- 选手 / 校友管理
- 赛事管理
- 团体管理
- 团体队员管理
- 赛事结果 Excel 上传
- 导入预览与错误修正
- 积分规则配置
- 远程参赛奖补录入
- 代表队参赛积分录入
- 办赛奖励录入
- 捐赠赞助积分录入
- 年度排行榜刷新
- 导入日志与审计记录

### 4.2.2 管理后台设计原则

- Excel 上传和导入预览优先适配桌面大屏。
- 所有导入数据必须先预览，管理员确认后再入库。

- 所有人工调整积分必须保留操作日志。
- 排行榜计算结果应可追溯到具体积分明细。

数据库推荐：

## 6.1 核心数据表

## 6.2 关键数据流

### 6.2.1 单项双打赛

```text
赛事结果 Excel
    ↓
event_results：记录冠军 / 亚军 / 前四名 / 前八名 / 参赛名单
    ↓
PointsGenerationService：根据赛事等级和组别查 points_rules
    ↓
entries_points：为每名搭档生成完整个人积分
    ↓
RankingService：汇总年度总积分
```

### 6.2.2 团体赛

```text
团体赛结果 Excel
    ↓
event_results：记录团队成绩和团队总积分
    ↓
team_members：记录有效队员名单
    ↓
PointsGenerationService：round(团队总积分 / 有效队员人数)
    ↓
entries_points：为每名有效队员生成团体分摊积分
    ↓
RankingService：汇总年度总积分
```

### 6.2.3 其他积分来源

```text
远程参赛 / 代表队参赛 / 办赛奖励 / 捐赠赞助
    ↓
管理员录入或 Excel 导入
    ↓
entries_points：生成对应积分明细
    ↓
RankingService：汇总年度总积分
```

## 7. 年度排行榜计算架构

系统不采用 52 周滚动周期，也不采用 Best-of-N。

年度排行榜计算逻辑如下：

```text
选择当前赛季
    ↓
读取当前赛季全部 entries_points
    ↓
按 player_id 分组
    ↓
SUM(points_earned) 得到个人年度总积分
    ↓
按年度总积分降序计算并列排名
    ↓
同积分页面展示按姓名首字母排序
```

排序规则：

```text
排名：RANK() OVER (ORDER BY total_points DESC)
展示：total_points DESC, name_initial ASC, full_name ASC, player_id ASC
```

说明：

- 总积分相同则名次并列。
- 同积分不再使用高级别赛事积分、参赛数量、单项最高分等规则打破名次。
- 姓名首字母排序只影响页面展示顺序，不影响并列名次。

## 8. Excel 上传与异步任务架构

## 8.1 上传流程

```text
管理员上传赛事结果 Excel
    ↓
后端保存 Excel 文件
    ↓
创建 uploads 记录
    ↓
返回“处理中”
    ↓
后台任务解析 Excel
    ↓
生成导入预览
    ↓
管理员确认
    ↓
写入 event_results / entries_points
    ↓
刷新年度排行榜
```

## 8.2 任务队列

推荐使用：

```text
Redis + ARQ
```

备选：

```text
Redis + Celery
```

后台任务包括：

- Excel 文件解析任务
- 选手名称匹配任务
- 赛事结果校验任务
- 积分明细生成任务
- 年度排行榜刷新任务
- AI 辅助解析任务，后期可选

## 9. AI 辅助架构

AI 在本系统中不是积分计算器，而是数据导入辅助工具。

## 9.1 AI 适合做的事情

- 非标准 Excel 表头识别
- 名次列识别
- 组别列识别
- 双打搭档识别
- 团体名称识别
- 团体队员名单识别
- 跨省 / 跨境奖补信息识别
- 选手姓名简称匹配
- 院系名称纠错
- 异常数据解释
- 导入预览辅助生成

## 9.2 AI 不应该做的事情

- 不负责最终积分计算。
- 不负责最终排名计算。
- 不直接写数据库。
- 不直接修改积分规则。
- 不跳过管理员确认。

## 9.3 AI 技术路径

### 第一阶段

不接入复杂 AI，采用标准 Excel 模板：

```text
标准 Excel 模板
openpyxl / pandas 解析
Pydantic 校验
管理员确认
确定性入库
```

### 第二阶段

接入 Dify，用于快速构建 AI Excel 解析工作流：

```text
Dify
LLM 节点
代码节点
HTTP API 节点
结构化 JSON 输出
```

### 第三阶段

接入 LangGraph，用于更复杂的 AI 工作流：

```text
LangGraph
状态机
解析节点
校验节点
纠错节点
人工确认节点
```

## 12. 架构总结

推荐系统架构为：

```text
微信小程序 / H5 用户端
+
Web 管理后台
+
FastAPI 后端服务
+
PostgreSQL 数据库
+

+

+
```

该架构既能满足当前 THA 年度赛事积分排名的核心需求，又能在后续扩展 AI 导入、历史排名、校庆报名、种子排序、数据统计和赛事运营分析等能力。
