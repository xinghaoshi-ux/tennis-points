# 人工智能驱动的网球赛事积分与排名系统需求与架构设计

## 1. 项目概述

本项目旨在构建一套面向清华校友网球协会 THA 年度赛事体系的积分与排名系统。系统围绕 2026-2027 年度巡回赛、校庆总决赛、分站赛、卫星赛、团体赛、代表队赛事、办赛奖励和捐赠赞助奖励建立统一的个人年度积分排名体系。

系统的核心路径为：管理员上传赛事结果 Excel，系统解析并校验赛事名次、组别、队员、团体和奖励信息，根据 THA 年度积分规则生成选手积分和排名，并在 Web 或微信小程序端展示年度排行榜、个人主页和积分明细。

该产品不是简单的“Excel 转排行榜”工具，而是一个以赛事事实数据为底座、以积分规则为核心、以排行榜和个人页为展示出口、未来可接入 AI 数据导入能力的网球竞技数据平台。

---

## 2. 产品定位

### 2.1 一句话定义

一个面向内部网球比赛的积分排名平台，管理员上传比赛结果 Excel，系统自动解析比赛数据，根据自定义积分规则计算选手积分与排名，并展示排行榜、个人积分明细和赛事数据。

### 2.2 目标用户

#### 2.2.1 普通用户 / 参赛者 / 观众

用户端不需要登录，定位为公共积分榜查看平台。普通用户、参赛者和观众进入系统后即可查看年度排行榜。

普通用户主要关注：

- 当前年度排行榜
- 选手当前排名
- 选手当前积分
- 点击选手名称查看积分构成
- 与其他选手的积分差距
- 所属院系或部门排名

#### 2.2.2 管理员 / 赛事组织者

管理员主要关注：

- 选手管理
- 赛事创建
- Excel 比赛结果上传
- 数据校验与修正
- 积分规则配置
- 排行榜发布
- 错误数据处理

#### 2.2.3 观众 / 社群成员

观众主要关注：

- 当前排行榜
- 选手个人战绩
- 院系或部门竞争情况
- 最新赛事结果
- 排名变化

## 3. 核心业务流程

系统主流程如下：

```text
管理员创建赛事
    ↓
设置赛季、赛事级别、组别和积分规则
    ↓
上传赛事结果 Excel
    ↓
系统解析 Excel
    ↓
校验选手、双打组合、团体队员、名次和奖励信息
    ↓
生成赛事结果记录
    ↓
根据名次、团体分摊和奖励规则计算积分
    ↓
刷新排行榜
    ↓
用户无需登录，进入公共积分榜查看排行榜，点击选手名称查看积分明细弹窗
```

| 赛事名称 | 赛季 | 赛事级别 | 团体名称 | 团体成绩 | 团队总积分 | 有效队员名单 |

### 4.4.2 数据校验规则

上传后需要校验：

- 赛事是否存在或是否允许自动创建
- 赛事是否属于当前赛季
- 赛事等级是否符合规则
- 组别是否合法
- 成绩是否合法，例如冠军、亚军、前四名、前八名、参赛
- 选手是否存在或是否需要新建
- 双打组合是否为两名选手
- 团体赛有效队员名单是否为空
- 团体赛分摊人数是否正确
- 远程参赛奖补是否与选手逐一对应
- 是否重复上传同一赛事同一成绩

上传后不应直接入库，而应展示导入预览。

| 行号 | 赛事 | 级别 | 组别 / 团体 | 成绩 | 选手 / 队员 | 预计积分 | 状态 |

| 1 | THA500 成都站 | THA500 | 甲组 | 冠军 | 张三 / 李四 | 500 / 人 | 正常 |
| 2 | THA A 团体赛北京站 | THA A | 计算机学院队 | 冠军 | 8 名队员 | 225 / 人 | 正常 |
| 3 | THA800 北京站 | THA800 | 女双组 | 前四名 | 小王 / 赵六 | 280 / 人 | 选手待匹配 |

管理员可以：

- 确认导入
- 修改选手匹配
- 忽略某行
- 下载错误报告

## 5. THA 年度积分规则与算法设计

> **规则确认版：根据《2026-2027 清华校友网球协会年度赛事总计划》调整。系统不再采用 52 周滚动周期，不采用 Best-of-N。排名按年度赛季统计，赛季内所有合规积分全部计入个人总积分。**

系统排名的公信力来自清晰、稳定、可解释的积分算法。当前积分系统服务于 THA 年度巡回赛、2027 校庆总决赛报名筛选、种子排序、年度积分奖励和校友贡献激励。

## 5.1 年度赛季积分模型

### 5.1.1 赛季定义

系统以年度赛季作为积分周期，而非 52 周滚动窗口。

本阶段建议将赛季定义为：

```text
2026-2027 赛季 = 自前一年度校庆赛起，至 2027 校庆总决赛前或总决赛结束止
```

赛季具体起止日期应在后台配置，例如：

| 字段 | 示例 |
|---|---|
| 赛季名称 | 2026-2027 THA 赛季 |
| 赛季开始日期 | 2026 校庆赛开始日 |
| 赛季结束日期 | 2027 校庆赛结束日 |
| 是否当前赛季 | 是 |

### 5.1.2 年度个人积分计算办法

个人年度总积分为当前赛季内所有合规积分之和。

```text
个人年度总积分 = 单项赛事积分 + 团体赛分摊积分 + 代表队参赛积分 + 办赛奖励积分 + 捐赠赞助积分 + 远程参赛奖补积分
```

系统不进行 Best-of-N 截取，所有比赛、活动和奖励积分均计入年度总积分。

### 5.1.3 积分用途

年度积分用于：

- 2027 校庆总决赛单项报名筛选
- 校庆总决赛种子排序
- 年度总积分排名前三奖励
- 校庆颁奖与校友贡献表彰
- 评估校友年度参赛活跃度和贡献度


这与团体赛“团队总积分平均分摊”不同。双打单项赛的每名搭档直接获得该成绩对应的完整个人积分。

## 5.3 THA 团体赛积分系统

### 5.3.1 赛事分类

团体赛按照赛事级别分为：

| 赛事等级 | 赛事类型 |
|---|---|
| THA S 级 | 校庆团体赛 |
| THA A 级 | 高级别团体赛 |
| THA B 级 | 标准团体赛 |

### 5.3.2 团体积分确认方式

团体赛积分根据以下因素确认：

- 团体赛级别
- 清华校友参赛人数规模
- 单场场次
- 团队最终成绩

个人积分为团队总积分除以有效队员人数后平均分配，并四舍五入取整数。

```text
队员个人积分 = round(团队总积分 / 有效队员人数)
```

### 5.3.3 THA S 级校庆团体赛积分

赛制：三场制。

| 团队成绩 | 团队总积分 |
|---|---:|
| 冠军 | 3000 |
| 亚军 | 1800 |
| 前四名 | 1200 |
| 参赛 | 600 |

### 5.3.4 THA A 级团体赛积分

赛制：三场制。

| 团队成绩 | 团队总积分 |
|---|---:|
| 冠军 | 1800 |
| 亚军 | 1200 |
| 前四名 | 800 |
| 参赛 | 200 |

识别标准：签表中参赛校友人数大于 40 人。

### 5.3.5 THA B 级团体赛积分

赛制：三场制。

| 团队成绩 | 团队总积分 |
|---|---:|
| 冠军 | 900 |
| 亚军 | 600 |
| 前四名 | 400 |
| 参赛 | 100 |

识别标准：签表中参赛校友人数为 16-40 人。

### 5.3.6 团体赛有效队员

系统应在团体赛结果导入时记录每支团队的有效队员名单。有效队员是团队总积分平均分配的依据。

第一版建议：

```text
团体赛有效队员 = 管理员在结果 Excel 中上传的团队队员名单
```

后续可扩展为：

```text
仅实际上场队员参与积分分摊
```

### 5.3.7 团体赛积分示例

某 THA A 级团体赛，计算机学院队获得冠军，团队总积分为 1800 分，有效队员 8 人。

```text
每名队员获得积分 = round(1800 / 8) = 225 分
```

个人页展示说明：

```text
你在“THA A 级团体赛北京站”中代表“计算机学院队”参赛。该队获得冠军，团队总积分 1800 分，有效队员 8 人，因此你获得 225 分团体赛分摊积分。
```

## 5.4 远程参赛奖补积分

系统需要支持跨境、跨省参赛的额外奖补积分。

| 类型 | 奖补积分 | 说明 |
|---|---:|---|
| 跨境参赛 | 500 | 每站额外发放 |
| 跨省参赛 | 200 | 每站额外发放 |

远程参赛奖补可以与单项赛事或团体赛事积分叠加。

例如：

```text
某选手参加 THA500 成都站并获得前四名 180 分，同时属于跨省参赛，则总计获得 180 + 200 = 380 分。
```

系统需要在积分明细中单独标识远程参赛奖补来源，避免用户误解为赛事成绩积分。

## 5.5 代表校友网球俱乐部网球队参赛积分

参加校际正式指定高校校友对抗赛，可获得对应积分。

当前指定比赛包括：

- 文促会
- 开源杯
- 翰林杯
- 全国高校校友网球锦标赛
- 其他经审议增补赛事

个人获得积分暂定如下：

| 积分类型 | 积分 |
|---|---:|
| 胜场积分 | 40 分 / 场 |
| 负场积分 | 20 分 / 场 |
| 每次组队参赛基础积分 | 20 分 |

示例：

```text
参加开源杯比赛，二胜一负：40 + 40 + 20 + 20 = 120 分
```

该类积分属于“代表队参赛积分”，直接计入年度个人总积分。

## 5.7 捐赠与赞助积分

为校友各网球队捐款捐物也可获得积分。

暂定规则：

```text
捐款捐物等值 10000 元，对应奖补 200 积分
```

系统应支持管理员录入捐赠金额或等值金额，并自动或手动计算对应积分。

## 5.8 总积分并列与展示排序规则

当多名选手年度总积分相同时，名次并列。

例如：

| 展示顺序 | 排名 | 姓名 | 积分 |
|---:|---:|---|---:|
| 1 | 1 | 张三 | 1000 |
| 2 | 1 | 李四 | 1000 |
| 3 | 3 | 王五 | 800 |

同积分选手不再通过高级别赛事积分、参赛数量或单项最高分打破名次。为了保证页面展示顺序稳定，同积分者仅按照姓名首字母排序。

排序规则：

```text
年度总积分 DESC
同积分并列排名
同积分展示顺序按姓名首字母 ASC
姓名首字母相同时按姓名 ASC
姓名仍相同时按 player_id ASC
```

## 7. 数据库设计

系统应采用关系型数据库，推荐 PostgreSQL。

不建议使用 MongoDB 等文档型数据库作为核心数据存储，因为本系统高度依赖：

- 关联查询
- 分组聚合
- 排名排序
- 时间窗口
- 窗口函数
- 数据一致性

---

## 7.1 核心设计原则

### 7.1.1 事实数据与衍生数据分离

不要在选手表中直接存储当前积分和当前排名。

原因是积分和排名依赖：

- 年度赛季
- 赛事结果名次
- 赛事等级
- 赛事组别
- 积分规则
- 远程参赛奖补
- 团体赛队员分摊
- 办赛、代表队参赛、捐赠赞助等额外积分

任何赛事结果修正、积分规则修正或额外积分补录，都可能导致年度排名变化。

因此：

```text
比赛记录 / 赛事积分 = 事实数据
排行榜 = 衍生计算结果
```

## 8. 年度排行榜计算逻辑

排行榜计算流程如下：

### 8.1 选择当前赛季

系统首先确定当前有效赛季，例如：

```text
2026-2027 THA 赛季
```

排行榜只统计该赛季内的 entries_points 记录。

### 8.2 获取赛季内所有积分记录

从 entries_points 中找到每位选手在当前赛季内的全部积分记录，包括：

- 单项双打赛事成绩积分
- 团体赛平均分摊积分
- 跨境 / 跨省远程参赛奖补积分
- 代表校友网球队参赛积分
- 办赛奖励积分
- 捐赠与赞助积分
- 人工调整积分

### 8.3 全部积分计入总分

系统不采用 Best-of-N。当前赛季内所有合规积分全部计入个人年度总积分。

```text
个人年度总积分 = SUM(当前赛季内该选手所有 entries_points.points_earned)
```

### 8.4 计算参赛数量与贡献数量

排行榜中的“参赛数量”建议展示当前赛季内获得赛事类积分的赛事数量。

建议区分：

| 概念 | 说明 |
|---|---|
| 参赛数量 | 当前赛季内参与的单项赛、团体赛、代表队赛事数量 |
| 积分记录数量 | 当前赛季内所有积分明细条数，包含奖励与赞助积分 |
| 办赛贡献数量 | 当前赛季承办或主办赛事次数 |

### 8.5 同积分并列排名

当多名选手年度总积分相同时，名次并列。

排名计算使用并列排名逻辑：

```text
RANK() OVER (ORDER BY total_points DESC)
```

例如两人并列第 1，则下一名为第 3。

### 8.6 页面展示顺序

同积分选手的名次相同，但页面需要稳定展示顺序。同积分者按姓名首字母排序。

```text
总积分 DESC
姓名首字母 ASC
姓名 ASC
player_id ASC
```

### 8.7 个人页积分明细

个人页应按照积分来源拆分展示：

| 来源类型 | 示例 |
|---|---|
| 单项赛事 | THA500 成都站，前四名，180 分 |
| 团体赛 | THA A 级团体赛冠军，团队 1800 分，8 人分摊，个人 225 分 |
| 远程参赛 | 跨省参赛奖补，200 分 |
| 代表队参赛 | 开源杯二胜一负，120 分 |
| 办赛奖励 | 承办 THA500 分站赛，500 分 |
| 捐赠赞助 | 捐赠等值 10000 元，200 分 |

## 9. AI 能力设计

AI 在本系统中的定位应当是“辅助数据录入与异常解释”，而不是“最终排名计算器”。

---

## 9.1 AI 适合做的事情

AI 适合处理：

- 非标准 Excel 表头识别
- 判断哪一列是赛事名次
- 判断哪一列是冠军、亚军、前四名、前八名、参赛名单
- 识别双打搭档组合
- 识别团体名称和队员名单
- 识别跨省 / 跨境参赛奖补信息
- 合并单元格处理
- 选手简称识别
- 院系名称纠错
- 异常数据解释
- 导入预览生成

---

## 9.2 AI 不应该做的事情

AI 不应负责：

- 最终排名计算
- 积分加总
- 同分排序
- 无审核地写入数据库
- 修改核心积分规则
- 决定最终数据是否有效

积分和排名必须由确定性代码完成，确保结果可复现、可解释、可审计。

---

## 9.3 AI 辅助导入流程

推荐流程：

```text
管理员上传赛事结果 Excel
    ↓
系统读取 Excel 基础结构
    ↓
AI 判断字段含义
    ↓
生成结构化 JSON
    ↓
后端用 Schema 严格校验
    ↓
发现异常则返回预览和错误
    ↓
管理员确认
    ↓
写入 event_results 表
    ↓
后端确定性算法生成 entries_points
    ↓
刷新年度排行榜
```

---

## 9.4 Dify 与 LangGraph 选择

### 9.4.1 Dify

适合：

- 快速搭建 AI Excel 解析流程
- 团队 AI 工程能力不强
- 希望通过可视化工作流配置提示词和节点
- 希望非技术人员参与调试工作流

建议用途：

```text
AI 辅助解析 Excel
AI 识别字段
AI 输出结构化 JSON
AI 生成错误说明
```

不建议 Dify 负责：

```text
数据库核心写入
积分计算
排名计算
权限系统
用户系统
```

### 9.4.2 LangGraph

适合：

- 需要高度可控的 AI 工作流
- 需要状态机、重试、分支、自我修复
- 有较强 Python 工程能力
- 希望把 AI 流程深度集成到 FastAPI 后端

LangGraph 更适合成熟阶段的自研 AI 导入引擎。

### 9.4.3 第一版建议

第一版建议先不用复杂 AI Agent 框架，而是采用：

```text
标准 Excel 模板
openpyxl / pandas 解析
Pydantic 校验
人工确认
确定性入库
自动计算排名
```

等真实用户使用后，再根据 Excel 混乱程度决定是否接入 AI。

## 10.2 前端技术选择

### 10.2.1 用户端

如果要同时支持 Web 和微信小程序，推荐：

```text
Uni-app + Vue 3 + TypeScript
```

优势：

- 一套代码支持 H5 和微信小程序
- 降低维护成本
- Vue 生态成熟
- 适合移动端排行榜和个人页展示

### 10.2.2 状态管理

推荐：

```text
Pinia
```

用于缓存：

- 当前用户信息
- 选手基础信息
- 排行榜筛选条件
- 页面间跳转状态

### 10.2.3 UI 组件库

推荐：

```text
Wot Design Uni
```

也可选：

```text
uView Plus
Vant Weapp
```

### 10.2.4 管理后台

管理后台建议使用：

```text
Vue 3 + Element Plus
```

原因：

- Excel 上传更适合桌面端
- 表格预览更适合大屏
- 积分规则配置更适合后台表单
- 错误日志和数据修正需要复杂交互

## 10.3 后端技术选择

推荐：

```text
FastAPI + Pydantic + SQLAlchemy 2.0
```

### 10.3.1 FastAPI 优势

- Python 生态适合 Excel 解析和 AI 集成
- 原生支持异步接口
- 自动生成 API 文档
- Pydantic 校验能力强
- 与 SQLAlchemy、Redis、任务队列集成方便

### 10.3.2 数据校验

Pydantic 应用于：

- Excel 解析结果校验
- API 请求参数校验
- 比赛结果结构校验
- 积分规则校验
- AI 输出 JSON 校验

## 10.4 数据库选择

推荐：

```text
PostgreSQL
```

原因：

- 支持复杂 SQL
- 支持窗口函数
- 支持视图和物化视图
- 支持事务一致性
- 支持 JSON 字段
- 适合排行榜、积分明细和统计分析

## 10.5 普通视图与物化视图

### 10.5.1 普通视图

适合早期：

- 数据量小
- 用户量小
- 排行榜实时计算压力不大

### 10.5.2 物化视图

适合后期：

- 排行榜访问频繁
- 比赛数据增多
- 需要更快查询速度

推荐演进方式：

```text
第一版：普通查询或普通视图
后续版本：上传赛事后刷新物化视图
```

## 10.6 后台任务架构

Excel 解析和排名刷新不应阻塞前端请求。

推荐流程：

```text
用户上传 Excel
    ↓
后端保存文件
    ↓
创建 upload 记录
    ↓
立即返回“处理中”
    ↓
后台任务解析 Excel
    ↓
生成预览结果
    ↓
管理员确认
    ↓
导入数据库
    ↓
后台刷新排行榜
```

推荐组件：

```text
Redis + ARQ
```

也可以选择：

```text
Redis + Celery
```

第一版建议 ARQ，轻量且适合 FastAPI 异步生态。

```text

```

### 12.2 MVP 用户端功能

- 排行榜

- 赛事列表

### 12.3 MVP 管理端功能

- 管理员登录

### 12.4 MVP 算法

- 年度赛季积分

- 远程参赛奖补积分计入年度总分
- 个人积分明细

- 多智能体 Agent
- 逐场比分与胜负明细录入
- 自动签表

## 15. 风险与边界

### 15.1 数据一致性风险

风险：比赛结果修正后排行榜未同步。

解决：排行榜基于事实数据动态计算，必要时使用物化视图刷新机制。

### 15.2 Excel 格式混乱风险

风险：用户上传格式不一致导致解析失败。

解决：第一版提供标准模板，后续再引入 AI 辅助解析。

### 15.3 AI 幻觉风险

风险：AI 错误理解数据或生成错误结果。

解决：AI 只做辅助解析，最终入库前必须经过 Schema 校验和管理员确认。

### 15.4 排名争议风险

风险：选手不理解排名结果。

解决：公开积分规则，并在个人页展示积分构成和同分排序原因。

### 15.5 技术复杂度风险

风险：第一版引入过多 AI、Agent、Elo、双打、签表功能，导致迟迟无法上线。

解决：MVP 聚焦标准 Excel、积分规则、排行榜和个人页。

## 18. 总结

本系统的核心不是简单地把 Excel 数据转换成排行榜，而是构建一个稳定、透明、可审计、可扩展的网球赛事积分与排名平台。

第一阶段应优先完成标准 Excel 导入、积分规则配置、排行榜生成和个人积分明细展示。AI 能力应作为后续增强模块，用于解决非标准赛事结果 Excel 解析、名称匹配、团体队员识别和异常数据解释等问题，而不应参与最终的积分和排名计算。

推荐以“微信小程序用户端 + Web 管理后台 + FastAPI 后端 + PostgreSQL 数据库”的方式启动项目，先做可用 MVP，再逐步扩展 AI、历史排名、院系榜和专业竞技分析能力。
