# 项目级规则

> 本文定义 THA 年度赛事积分系统的项目级行为约束。
>
> 这些规则适用于整个项目生命周期，尤其适用于 Claude Code、协作开发和阶段性交付。

## 1. 项目级原则

### 1.1 复杂任务先输出计划，待确认后再执行

当任务满足以下任一条件时，必须先给出计划，再等待确认：

- 跨多个目录或多个子系统
- 需要较强技术取舍
- 影响架构、数据模型或 API 契约
- 可能引入较大范围文件改动
- 需求描述存在歧义或前置信息不足

执行要求：

1. 先输出分步骤计划
2. 明确待确认项
3. 等待用户确认或补充信息
4. 再进入实际实现

### 1.2 先文档后代码

项目必须遵循“先文档后代码”的顺序。

含义是：

- 先完成需求、设计、契约与方案类文档
- 再开始实现前端或后端代码
- 代码实现必须引用已确认的上游文档
- 文档未定稿前，不得提前固化业务实现

### 1.3 每个阶段必须有明确输出物

每个阶段必须以可检查、可保存、可回溯的输出物收口。

例如：

- 调研阶段：研究记录、分析结论
- 设计阶段：PRD、架构、数据模型、API 规范
- 开发阶段：前端实现、后端实现、测试结果
- 联调阶段：联调记录、问题清单
- 发布阶段：部署文档、上线清单、回滚方案

### 1.4 收口文档禁止擅自发散

收口文档的职责是“确认和固定”，不是“自由扩写”。

禁止行为包括：

- 在已收口文档中加入未经确认的新需求
- 擅自扩展范围外能力
- 把临时设想写成既定事实
- 用模糊表述掩盖未确认内容

### 1.5 前端不得伪造后端真实能力

前端实现必须尊重后端实际能力与契约。

禁止行为包括：

- 用假数据替代真实接口并冒充完成
- 在前端本地拼接业务结果作为最终答案
- 通过静态逻辑伪装后端已经支持的能力
- 绕过接口契约直接假定响应结构

前端可以做：

- 页面骨架与交互
- 接口联调前的占位展示
- 明确标识的 mock 调试

但 mock 只能用于开发验证，不能冒充正式能力。

### 1.6 后端不得偏离 API 契约

后端实现必须严格遵守已确认的 API 契约。

禁止行为包括：

- 随意修改请求字段
- 随意修改响应结构
- 临时增删字段而不更新文档
- 为了方便实现改变接口语义
- 在前端不知情的情况下变更错误码或状态含义

如果契约需要变更，必须先更新 `docs/08_api_spec/`，再同步实现。

### 1.7 主链路优先，MVP 优先

第一阶段必须优先保证主链路可用。

主链路优先意味着：

- 先完成核心流程
- 先保证关键用户场景可跑通
- 先打通最小闭环

MVP 优先意味着：

- 先做最小可用版本
- 再做增强能力
- 避免一开始追求“大而全”

## 2. 项目执行规则

### 2.1 文档驱动实现

所有实现必须基于 `docs/` 中已确认的上游文档。

优先级建议为：

1. 业务研究与问题定义
2. PRD
3. 架构与数据模型
4. API 契约
5. 前端计划与 UI 设计
6. 集成与部署文档
7. 代码实现

### 2.2 变更必须可追踪

任何影响结构、接口、数据模型、流程的变更，都必须：

- 有文档依据
- 有明确说明
- 有对应阶段输出物
- 需要时同步更新相关文档

### 2.3 不做无依据扩展

没有文档依据的内容，不得默认加入项目范围。

如果存在不确定性，应当：

- 列出待确认项
- 暂缓实现
- 等待上游文档确认

### 2.4 版本与契约稳定

在项目推进过程中，以下内容应尽量稳定：

- 目录结构
- 文档目录职责
- API 契约
- 数据模型核心字段
- 技术选型

如需变化，应先评估影响再修改。

## 3. 阶段性输出规则

### 3.1 00_setup

输出物应包括：

- 环境配置方案
- 工具检查清单
- 依赖策略
- 项目结构说明
- 项目规则

### 3.2 01_business_research 至 05_prd

输出物应逐步固化：

- 业务背景
- 问题定义
- 交互流程
- PRD

### 3.3 06_architecture 至 08_api_spec

输出物应固化：

- 系统架构
- 数据模型
- API 契约

### 3.4 09_frontend_plan 至 10_ui_design

输出物应固化：

- 前端拆分方案
- 页面与组件规划
- UI 规范

### 3.5 11_integration 至 12_release_deployment

输出物应固化：

- 联调清单
- 集成问题记录
- 发布部署方案
- 回滚与验收说明

## 4. 对 Claude Code 的执行约束

Claude Code 在本项目中应遵守以下行为：

- 复杂任务先给计划
- 未确认前不直接扩散实现
- 先产出文档，再产出代码
- 所有实现必须依据上游文档
- 任何 API 改动都必须先对齐契约
- 前端不能伪造后端能力
- 后端不能擅改接口契约
- 每阶段必须有收口输出
- 主链路和 MVP 优先于非核心增强

## 5. 待确认项

以下规则实施细节可在后续补充：

1. 哪些类型的任务必须强制输出计划
2. 计划的标准模板是否需要统一
3. 文档收口的审批方式
4. API 契约变更的审批流程
5. 每阶段输出物是否需要固定命名规范
6. MVP 的具体主链路范围

## 6. 结论

本项目规则的核心是“文档驱动、契约优先、阶段收口、主链路优先”。

只要这些规则稳定，项目在后续研究、设计、开发、联调和发布过程中就能保持一致的执行方式。
