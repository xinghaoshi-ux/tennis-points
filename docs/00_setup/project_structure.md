# 项目目录结构说明

> 本文用于说明项目根目录结构、各目录职责边界以及 `docs/` 下各阶段目录的用途。
>
> 本阶段仅做项目初始化与规范搭建，不涉及任何业务实现。

## 1. 项目目录结构总览

建议项目采用以下标准结构：

```text
project-root/
├─ frontend/
├─ backend/
├─ docs/
│  ├─ 00_setup/
│  ├─ 01_business_research/
│  ├─ 02_competitive_analysis/
│  ├─ 03_problem_modeling/
│  ├─ 04_interaction_design/
│  ├─ 05_prd/
│  ├─ 06_architecture/
│  ├─ 07_data_model/
│  ├─ 08_api_spec/
│  ├─ 09_frontend_plan/
│  ├─ 10_ui_design/
│  ├─ 11_integration/
│  └─ 12_release_deployment/
├─ CLAUDE.md
└─ README.md
```

## 2. 各顶层目录用途

### 2.1 `frontend/`

`frontend/` 用于承载所有前端相关内容，包括但不限于：

- 用户端页面
- 管理端页面
- 前端路由与状态管理
- 前端组件、样式与静态资源
- 前端构建、测试、打包与发布配置

该目录只负责前端实现，不承担业务文档编写职责。

### 2.2 `backend/`

`backend/` 用于承载所有后端相关内容，包括但不限于：

- API 服务
- 领域服务与业务逻辑
- 数据库访问与迁移
- 权限、认证与鉴权
- 文件处理、任务处理与集成能力
- 后端测试与运行配置

该目录只负责后端实现，不承担阶段性业务文档编写职责。

### 2.3 `docs/`

`docs/` 用于承载项目从研究、设计到交付的全部过程文档，是项目的主导依据之一。

该目录应优先于代码存在和演进，所有实现都应以 `docs/` 中已确认的上游文档为依据。

### 2.4 `CLAUDE.md`

`CLAUDE.md` 是项目级行为约束文件，用于规范 Claude Code 在整个项目中的工作方式。

它的作用是：

- 约束任务执行顺序
- 明确文档优先级
- 约束实现边界
- 控制阶段输出物
- 限制不基于文档的自由发散

### 2.5 `README.md`

`README.md` 是项目入口说明文档，用于让新成员快速理解：

- 项目目标
- 目录结构
- 文档阅读顺序
- 当前阶段
- 下一步应该先看哪些文档

## 3. `frontend/`、`backend/`、`docs/` 的职责边界

### 3.1 `frontend/` 的职责边界

`frontend/` 只做“呈现与交互”，不做真实后端能力的伪造或越权实现。

允许承担：

- 页面展示
- 表单交互
- 用户输入校验
- API 调用
- 结果渲染
- 前端状态管理

不应承担：

- 伪造后端返回结果
- 自行模拟真实业务规则作为最终实现
- 绕过后端 API 契约直接拼接业务结论
- 把后端应负责的数据校验和权限控制放到前端替代

### 3.2 `backend/` 的职责边界

`backend/` 只做“业务真实能力与契约提供”，不应偏离上游文档定义的 API 契约。

允许承担：

- 业务规则实现
- API 输出定义
- 数据持久化
- 权限控制
- 外部服务集成
- 异步任务和系统集成

不应承担：

- 擅自修改接口语义
- 未经文档确认改变请求响应结构
- 临时增加前端依赖的隐式字段而不更新文档
- 为了方便实现而破坏契约稳定性

### 3.3 `docs/` 的职责边界

`docs/` 是项目规则、调研、设计与交付的事实来源。

允许承担：

- 需求分析
- 竞品分析
- 问题定义
- 交互设计
- PRD
- 架构设计
- 数据模型
- API 规范
- 前端计划
- UI 设计
- 集成说明
- 发布部署说明

不应承担：

- 业务代码
- 运行时逻辑
- 临时个人笔记代替正式文档
- 未确认的随意发散内容

## 4. `docs/` 下各子目录用途

### 4.1 `docs/00_setup/`

用于基础环境、依赖策略、工具清单、项目结构和项目规则等初始化内容。

典型输出物：

- 环境配置方案
- 工具检查清单
- 依赖策略
- 项目结构说明
- 项目规则

### 4.2 `docs/01_business_research/`

用于业务研究与业务背景收集。

内容重点：

- 业务目标
- 用户角色
- 赛事积分场景
- 业务约束
- 现有流程

### 4.3 `docs/02_competitive_analysis/`

用于竞品分析和方案对比。

内容重点：

- 同类产品能力比较
- 方案优缺点
- 可借鉴点
- 适用边界

### 4.4 `docs/03_problem_modeling/`

用于问题建模，把业务问题抽象成可设计、可实现、可验证的对象。

内容重点：

- 核心问题
- 业务对象
- 状态流转
- 规则边界
- 风险点

### 4.5 `docs/04_interaction_design/`

用于交互设计，定义用户如何使用系统。

内容重点：

- 关键流程
- 页面跳转
- 表单交互
- 错误提示
- 状态反馈

### 4.6 `docs/05_prd/`

用于产品需求文档沉淀。

内容重点：

- 功能范围
- 优先级
- 需求说明
- 验收标准
- 非功能要求

### 4.7 `docs/06_architecture/`

用于架构设计说明。

内容重点：

- 系统架构
- 模块划分
- 技术选型
- 服务边界
- 部署拓扑

### 4.8 `docs/07_data_model/`

用于数据模型设计。

内容重点：

- 核心实体
- 表结构
- 字段定义
- 关系说明
- 索引与约束

### 4.9 `docs/08_api_spec/`

用于 API 契约与接口规范。

内容重点：

- 接口清单
- 请求响应结构
- 状态码
- 鉴权方式
- 错误码

### 4.10 `docs/09_frontend_plan/`

用于前端实施计划。

内容重点：

- 页面拆分
- 组件规划
- 路由规划
- 前端状态流转
- 与 API 的对接范围

### 4.11 `docs/10_ui_design/`

用于 UI 设计说明与视觉规范。

内容重点：

- 页面原型
- 视觉规范
- 组件样式
- 响应式规则
- 交互状态展示

### 4.12 `docs/11_integration/`

用于联调、集成和验证说明。

内容重点：

- 前后端联调流程
- 外部服务接入方式
- 联调顺序
- 验证清单
- 常见问题记录

### 4.13 `docs/12_release_deployment/`

用于发布、部署和上线说明。

内容重点：

- 发布流程
- 环境配置
- 部署步骤
- 回滚策略
- 上线检查

## 5. 项目级规则摘要

项目结构必须服务于以下规则：

- 复杂任务先输出计划，待确认后再执行
- 先文档后代码
- 每个阶段必须有明确输出物
- 收口文档禁止擅自发散
- 前端不得伪造后端真实能力
- 后端不得偏离 API 契约
- 主链路优先，MVP 优先

## 6. 待确认项

以下结构性问题在当前阶段建议明确：

1. `frontend/` 是否需要同时容纳 H5、Web 管理端和小程序多端实现
2. `backend/` 是否采用单体或模块化单体作为第一阶段实现方式
3. `docs/` 是否需要按阶段增加编号之外的专题目录
4. 是否需要为根目录补充 `.gitignore`、`.env.example` 等基础文件
5. 是否需要在 `docs/00_setup/` 下再增加模板类文档

## 7. 结论

该目录结构的核心价值在于：

- 把“研究、设计、实现、发布”分层
- 把“文档、前端、后端”分工
- 把“上游约束、实现过程、交付结果”串起来

当目录职责清晰后，后续每个阶段的工作边界、输出物和验收标准都会更容易控制。
