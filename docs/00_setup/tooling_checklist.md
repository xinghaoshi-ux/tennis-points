# 本地工具检查清单

> 适用范围：THA 年度赛事积分系统开发前的本地环境准备。
>
> 本文只列出工具检查项和判定标准，不包含任何业务实现。

## 1. 检查目标

在开始开发前，确保开发者本地具备完成以下工作的条件：

- 启动前端项目
- 启动后端项目
- 连接数据库和缓存
- 执行测试与迁移
- 调试环境变量
- 调用外部 AI / LLM / 文件存储服务（如项目阶段需要）

## 2. 通用工具检查

### 2.1 必备工具

| 工具 | 用途 | 检查方式 | 通过标准 |
| --- | --- | --- | --- |
| Git | 版本管理 | `git --version` | 输出版本号 |
| 代码编辑器 | 代码编辑与调试 | 手动确认 | 可以正常打开项目 |
| 终端工具 | 执行命令 | 手动确认 | 可正常输入命令并查看结果 |
| 网络连接 | 拉取依赖、访问外部服务 | 访问公共站点或依赖源 | 能正常联网 |

### 2.2 推荐工具

| 工具 | 用途 | 说明 |
| --- | --- | --- |
| Docker / Docker Desktop | 本地服务容器化 | 用于 PostgreSQL、Redis 等基础服务 |
| Postman / Insomnia | 接口调试 | 验证后端健康检查与基础 API |
| 浏览器 | 页面预览 | 调试 Web 前端 |
| 微信开发者工具 | 小程序调试 | 如用户端包含小程序形态则必需 |

## 3. 前端工具检查

### 3.1 Node.js 与包管理器

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| Node.js | `node -v` | 版本可输出，且符合项目约定 |
| 包管理器 | `npm -v` / `pnpm -v` / `yarn -v` | 仅保留项目约定的一种主方案 |
| 锁文件 | 检查仓库中的锁文件 | 仓库中只有一种锁文件策略 |

### 3.2 前端开发依赖

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| 前端依赖安装 | 执行安装命令 | 可成功安装，无冲突报错 |
| TypeScript | 编译检查 | 能通过基础类型检查 |
| ESLint / Prettier | 执行校验命令 | 能完成静态检查与格式化 |
| 构建工具 | 启动开发服务 | 项目能正常启动 |

### 3.3 用户端相关工具

如果用户端包含微信小程序：

- 需要安装微信开发者工具
- 需要在工具中导入项目
- 需要确保小程序开发配置已准备好
- 需要确认 H5 与小程序的运行模式是否一致

### 3.4 管理端相关工具

管理后台通常为 Web 项目，建议重点检查：

- 浏览器兼容性
- 大屏预览效果
- 构建脚本是否可执行
- 热更新是否正常

## 4. 后端工具检查

### 4.1 Python 运行环境

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| Python | `python --version` 或 `python3 --version` | 版本可输出，且符合项目约定 |
| 虚拟环境工具 | `venv`、Poetry 或 uv 之一 | 可创建并激活虚拟环境 |
| pip / 等价安装工具 | `pip --version` | 可用且与当前 Python 匹配 |

### 4.2 后端依赖与开发工具

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| 依赖安装 | 安装后端依赖 | 安装成功，无核心冲突 |
| FastAPI 启动 | 启动后端服务 | 服务正常监听端口 |
| 测试框架 | 执行测试命令 | 测试可以运行 |
| 迁移工具 | 执行迁移命令 | 能识别迁移配置并正常运行 |

### 4.3 后端健康检查

建议在开发初期准备健康检查相关工具：

- 浏览器或接口调试工具
- curl 或等价命令行请求工具
- 日志查看工具

通过标准：

- 健康检查接口可访问
- 返回格式稳定
- 能区分应用异常与依赖异常

## 5. 数据基础服务检查

### 5.1 PostgreSQL

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| PostgreSQL 客户端 | `psql --version` | 能输出版本 |
| 数据库服务 | 连接数据库 | 可成功登录或建立连接 |
| 数据库创建权限 | 创建测试库 | 可创建测试数据库 |
| 端口占用 | 检查默认端口 | 目标端口未被冲突占用 |

### 5.2 Redis

| 项目 | 检查方式 | 通过标准 |
| --- | --- | --- |
| Redis 客户端 | `redis-cli --version` | 能输出版本 |
| Redis 服务 | 连接测试 | 可执行 ping 并返回正常结果 |
| 端口占用 | 检查默认端口 | 目标端口未冲突 |

### 5.3 文件存储

如果项目阶段使用本地文件存储：

- 文件目录必须可写
- 上传目录必须存在或可自动创建
- 临时文件与正式文件目录要分离

如果项目阶段使用对象存储：

- 必须具备可用的桶或空间
- 本地开发环境需要有效凭证
- 上传、下载、删除最小能力要验证通过

## 6. AI / LLM / 外部服务工具检查

### 6.1 模型服务

如果当前阶段需要接入模型服务，应先检查：

- API Key 是否可用
- 基础 URL 是否正确
- 模型名称是否可访问
- 访问是否受网络或白名单限制
- 超时设置是否合理

### 6.2 MCP 服务

如果需要通过 MCP 接入外部能力，应先检查：

- MCP 服务地址可达
- 认证方式已配置
- 服务能力列表明确
- 是否允许本地调试环境访问
- 返回格式是否稳定

### 6.3 第三方平台

如果项目要接入 Dify、DeepSeek、COS、OSS 或其他平台，应先准备：

- 账号权限
- 访问密钥
- 回调地址
- 测试环境配置
- 费用与额度说明

## 7. 工具版本与兼容性检查

建议在项目启动前统一确认以下版本信息：

| 类别 | 需要确认的内容 |
| --- | --- |
| Node.js | 主版本是否固定 |
| 包管理器 | 团队是否统一 |
| Python | 是否与后端依赖兼容 |
| PostgreSQL | 版本是否支持当前驱动与迁移工具 |
| Redis | 版本是否兼容当前客户端 |
| Docker | 是否与本地系统兼容 |

## 8. 本地配置文件检查

### 8.1 必须存在的基础文件

| 文件 | 用途 | 说明 |
| --- | --- | --- |
| `.env.example` | 环境变量模板 | 仅包含字段名与占位值 |
| `.gitignore` | 忽略敏感文件 | 防止真实密钥提交仓库 |
| 依赖声明文件 | 记录安装依赖 | 前后端各自独立维护 |
| 锁文件 | 锁定依赖版本 | 保障安装可复现 |

### 8.2 本地个人文件

以下文件可存在于本地，但不应提交：

- `.env.local`
- 本地调试配置
- 个人证书或密钥
- 私有 API 测试配置

## 9. 开发前故障排查顺序

若本地环境无法启动，建议按以下顺序排查：

1. 检查工具版本是否正确
2. 检查依赖是否安装成功
3. 检查环境变量是否完整
4. 检查数据库和 Redis 是否可连
5. 检查端口是否被占用
6. 检查本地网络是否可访问外部依赖源
7. 检查是否存在锁文件冲突
8. 检查日志中是否有明确报错

## 10. 最小通过标准

当以下项全部满足时，可认为本地工具准备基本完成：

- [ ] Git 可用
- [ ] Node.js 可用
- [ ] 包管理器可用且已统一
- [ ] Python 可用
- [ ] 虚拟环境可创建
- [ ] PostgreSQL 可连接
- [ ] Redis 可连接
- [ ] Docker 可用，如项目需要
- [ ] 浏览器可用
- [ ] 微信开发者工具可用，如项目需要
- [ ] `.env.example` 与本地 `.env.local` 已准备
- [ ] AI / 外部服务凭证已准备，如项目阶段需要

## 11. 待确认项

以下事项建议在团队内确认后再开始正式开发：

1. Node.js 主版本
2. 前端统一包管理器
3. Python 主版本
4. 后端依赖管理工具
5. PostgreSQL 部署方式：本地安装还是 Docker
6. Redis 部署方式：本地安装还是 Docker
7. 是否必须安装微信开发者工具
8. 是否要求所有开发机都配置 Docker
9. 是否需要统一接口调试工具
10. AI / 外部服务是否在当前阶段就必须可用

## 12. 结论

工具检查的目的不是“列清单”，而是保证每个开发者在开始编码前都处于可复现的基础状态。

只要本地工具、依赖、服务和配置都通过最小验证，后续前后端开发、联调与部署的成本都会明显降低。
