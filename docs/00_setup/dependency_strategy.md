# 依赖管理策略

> 适用范围：THA 年度赛事积分系统的前端、后端以及基础服务依赖策略。
>
> 本文只讨论依赖治理方式，不包含任何业务实现。

## 1. 目标

依赖管理的目标不是“安装更多包”，而是保证：

- 安装可复现
- 版本可追踪
- 升级可控
- 多人协作不冲突
- 本地、测试、生产环境行为尽量一致

## 2. 当前项目的依赖管理原则

### 2.1 统一原则

建议整个项目遵循以下原则：

- 前端与后端分别维护独立依赖文件
- 同一子项目只保留一套主依赖管理方案
- 锁文件必须纳入版本控制
- 依赖升级必须记录原因和影响范围
- 不在业务代码中硬编码外部服务凭证

### 2.2 最小化原则

- 只引入当前阶段真正需要的依赖
- 不提前堆叠“未来可能会用”的包
- AI、MCP、对象存储、任务队列等能力按阶段引入
- 先跑通基础环境，再逐步扩展能力

### 2.3 可替换原则

如果后续技术栈发生调整，依赖管理策略也应同步更新，避免依赖文件与实际技术选型脱节。

## 3. 前端依赖管理建议

### 3.1 前端依赖边界

文档中的前端形态包括：

- 用户端：公共排行榜展示，可能使用 Uni-app + Vue 3 + TypeScript
- 管理端：Vue 3 + Element Plus + TypeScript

因此前端依赖应按以下层级划分：

- 运行时依赖：页面运行所需
- 开发依赖：构建、检查、测试、格式化所需
- 可选依赖：仅在启用某些能力时加入

### 3.2 前端包管理器建议

建议团队只选择一个主包管理器，避免混用导致锁文件冲突和安装差异。

可选项包括：

- `pnpm`
- `npm`
- `yarn`

建议在选型确认后，仓库内只保留对应锁文件，不要同时存在多种锁文件作为主方案。

### 3.3 前端依赖锁定建议

- 锁定主框架版本
- 锁定构建工具版本
- 锁定 UI 组件库版本
- 锁定状态管理和路由相关版本
- 锁定代码检查工具版本

### 3.4 前端依赖分层建议

| 层级 | 内容 | 说明 |
| --- | --- | --- |
| 基础框架层 | Vue、Uni-app 等 | 决定项目运行基础 |
| UI 层 | 组件库 | 决定界面实现方式 |
| 状态层 | Pinia 等 | 管理页面状态 |
| 请求层 | 统一请求封装相关依赖 | 调用后端接口 |
| 规范层 | TypeScript、ESLint、Prettier | 保证质量与一致性 |
| 测试层 | 单测、组件测试工具 | 按阶段引入 |

### 3.5 前端升级策略

- 小版本升级先在分支验证
- 主版本升级必须先做兼容性评估
- 构建失败时优先检查锁文件与 Node 版本
- 依赖变更要同步记录到变更说明中

### 3.6 前端依赖的待确认项

以下内容当前未完全锁定：

1. 用户端是否最终采用 Uni-app 还是纯 Vue Web
2. 管理端是否独立仓库还是同仓库多应用
3. 包管理器最终选型
4. 是否统一使用 monorepo 管理前后端前端项目
5. 是否引入前端单测框架作为第一阶段必选项

## 4. 后端依赖管理建议

### 4.1 后端依赖边界

后端建议采用 Python 技术栈，并围绕以下能力组织依赖：

- Web API
- 数据校验
- ORM / 数据库访问
- 数据库迁移
- 测试
- Redis / 任务处理
- Excel 解析
- AI 接入适配

### 4.2 Python 依赖管理方式

建议团队统一一种 Python 依赖管理方式，不要混用多套安装体系。

可选方向：

- `venv + pip`
- Poetry
- uv

无论采用哪种方式，都应满足以下要求：

- 依赖可复现
- 版本可锁定
- 安装命令统一
- CI 与本地一致

### 4.3 后端依赖分层建议

| 层级 | 内容 | 说明 |
| --- | --- | --- |
| Web 框架层 | FastAPI | 提供 API 服务 |
| 数据校验层 | Pydantic | 校验请求和结构化输出 |
| ORM 层 | SQLAlchemy 2.0 | 数据访问与模型管理 |
| 迁移层 | Alembic | 数据库结构版本管理 |
| 缓存层 | Redis 客户端 | 缓存与任务支持 |
| 测试层 | pytest | 单元测试与集成测试 |
| 解析层 | Excel 解析库 | 导入文件处理 |
| AI 接入层 | 模型 SDK、HTTP 客户端 | 仅在需要时引入 |

### 4.4 后端升级策略

- Python 版本变更前先确认兼容性
- 数据库驱动、ORM 和迁移工具需联动评估
- AI SDK 只在确有接入需求后加入
- Redis 客户端与任务队列工具需保证兼容

### 4.5 后端依赖的待确认项

当前需要明确的技术决策包括：

1. Python 主版本
2. 后端依赖管理工具最终选型
3. 是否标准化使用 Docker 作为开发环境
4. Excel 解析依赖是否只用标准库 + 通用解析库
5. AI 接入是否第一阶段就纳入后端依赖

## 5. 依赖文件建议

### 5.1 前端建议文件

根据最终选型，前端项目应至少维护以下文件类型：

- `package.json`
- 锁文件：`package-lock.json`、`pnpm-lock.yaml` 或 `yarn.lock`
- TypeScript 配置文件
- ESLint / Prettier 配置文件
- `.env.example`

### 5.2 后端建议文件

后端项目应至少维护以下文件类型：

- Python 依赖声明文件，如 `pyproject.toml`
- 锁定文件，如适用
- `.env.example`
- Alembic 配置文件
- 测试配置文件

### 5.3 仓库治理建议

- 依赖文件与锁文件必须提交版本控制
- 本地临时配置不得提交
- 不同子项目的依赖文件应彼此独立
- 依赖变更要配合代码审查

## 6. 环境变量与依赖的协同关系

依赖管理与环境变量管理需要配合：

- 依赖决定“代码能不能运行”
- 环境变量决定“连接哪个环境、使用哪个服务”

建议将以下信息分开管理：

- 依赖版本：写在依赖文件和锁文件中
- 服务地址：写在环境变量中
- 凭证密钥：写在环境变量或 Secret 系统中
- 部署参数：写在环境变量、部署脚本或平台配置中

## 7. AI / LLM / MCP / 外部服务的依赖接入建议

### 7.1 AI / LLM 依赖

AI 相关依赖不建议在基础阶段一次性全部装齐。

建议按以下步骤引入：

1. 先完成本地与核心服务环境
2. 再确认是否需要模型服务
3. 再接入对应 SDK 或 HTTP 客户端
4. 再实现结构化输出校验
5. 再加入日志、限流和降级逻辑

### 7.2 MCP 依赖

如后续使用 MCP，建议只为明确已确认的服务安装对应客户端或编排组件。

接入前应先明确：

- 目标 MCP 服务是否稳定
- 认证方式是否明确
- 是否需要本地代理
- 是否涉及额外网络权限

### 7.3 外部服务依赖

对象存储、第三方模型平台、消息服务等外部能力，都应该被视为“可插拔依赖”，而不是基础环境的默认前提。

建议：

- 先用本地文件系统或模拟服务跑通流程
- 再切换到真实外部服务
- 再验证异常处理与降级机制

## 8. 依赖检查策略

建议每次新增或升级依赖时执行以下检查：

- 安装是否成功
- 项目是否能启动
- 测试是否通过
- 静态检查是否通过
- 是否影响现有功能
- 是否引入额外安全风险

## 9. 最小依赖稳定标准

满足以下条件时，可认为依赖策略基础可用：

- [ ] 前端仅使用一套包管理器
- [ ] 前端锁文件唯一且有效
- [ ] 后端使用统一的 Python 依赖管理方式
- [ ] 后端依赖文件可复现安装
- [ ] 关键基础依赖版本已明确
- [ ] AI / 外部服务依赖按需引入
- [ ] 敏感凭证不写入依赖文件
- [ ] 依赖升级流程已约定

## 10. 待确认项汇总

以下决策建议在正式开发前明确：

1. 前端主包管理器选型
2. 后端 Python 依赖管理工具选型
3. 前端是否采用 monorepo
4. 后端是否采用 Poetry 或 uv
5. AI 相关 SDK 是否纳入基础依赖
6. MCP 是否在 V1 中启用
7. 文件存储是否依赖云服务
8. Docker 是否作为统一开发环境

## 11. 结论

依赖策略的核心是“统一、可复现、可升级、可审计”。

当包管理器、锁文件、Python 依赖和外部服务接入规则都明确后，后续开发、联调和部署都会更稳定，排障成本也会明显降低。
