# 环境配置方案

> 适用范围：THA 年度赛事积分系统的基础环境准备阶段。
>
> 本文只描述运行环境、依赖管理、环境变量、接入前准备和最小验证清单，不涉及任何业务实现。

## 1. 目标与原则

### 1.1 目标

在进入业务开发前，先确保项目具备稳定、可重复、可协作的基础运行环境，满足以下条件：

- 前端项目可以独立启动
- 后端项目可以独立启动
- 数据库、缓存、文件存储等基础服务可访问
- 环境变量可在本地、测试、生产之间安全区分
- AI / LLM / 外部服务接入点有明确的准备标准
- 团队成员可以按照统一方式完成本地环境初始化

### 1.2 基本原则

- 先约定环境，再写业务代码
- 先统一依赖管理，再统一开发工具
- 敏感信息不入库、不提交仓库
- 本地开发、测试、生产环境变量严格隔离
- 业务实现依赖的外部能力，在接入前必须完成可用性确认

## 2. 当前项目所需基础运行环境

根据《网球赛事积分与排名系统需求与架构设计》和《THA 年度赛事积分系统 Phase 开发计划》，项目建议至少准备以下运行环境。

### 2.1 前端运行环境

项目规划包含两个前端：

- 用户端：公共排行榜展示，可能使用 Uni-app + Vue 3 + TypeScript
- 管理端：Web 后台，可能使用 Vue 3 + TypeScript + Element Plus

因此本地前端环境建议准备：

- Node.js
- 包管理器（建议统一一种）
- 浏览器（用于 Web 预览）
- 微信开发者工具，如用户端包含微信小程序形态
- Git
- 统一的前端代码格式化与校验工具

### 2.2 后端运行环境

后端规划为 Python 技术栈，建议准备：

- Python
- 虚拟环境管理工具
- PostgreSQL
- Redis
- 文件存储能力（本地目录或对象存储）
- 测试框架与迁移工具

### 2.3 数据与基础服务环境

系统的核心依赖包括：

- PostgreSQL：存储赛季、选手、赛事、积分、导入记录等核心数据
- Redis：缓存、异步任务、导入解析任务或排行榜刷新任务
- 文件存储：保存 Excel 原文件、导入报告、附件等

### 2.4 AI / LLM / 外部服务环境

文档中明确提到未来可能接入：

- DeepSeek 或其他 LLM API
- Dify 或 LangGraph
- 对象存储服务，如腾讯云 COS、阿里云 OSS

这些能力在接入前应准备：

- API Key 或访问凭证
- 网络连通性确认
- 调用频率与费用预估
- 调试方式和降级策略

## 3. 推荐的基础技术环境

以下为基于文档内容整理的推荐项，不代表最终锁定方案；如后续技术选型调整，应同步更新本文档。

### 3.1 前端建议环境

| 项目 | 建议 | 说明 |
| --- | --- | --- |
| Node.js | 建议使用 LTS 版本 | 前端构建、依赖安装、代码检查都依赖 Node.js |
| 包管理器 | 建议统一使用 pnpm 或 npm 其一 | 避免团队内锁文件混乱 |
| 浏览器 | Chrome / Edge | 用于 Web 预览与调试 |
| 微信开发者工具 | 如存在小程序端则必备 | 用户端若包含小程序，需要本地调试环境 |
| Git | 必备 | 代码协作、分支管理、版本追踪 |

### 3.2 后端建议环境

| 项目 | 建议 | 说明 |
| --- | --- | --- |
| Python | 建议使用项目约定版本 | 与 FastAPI、SQLAlchemy、Pydantic 兼容 |
| 虚拟环境 | `venv`、Poetry 或 uv 其一 | 统一依赖隔离方式 |
| PostgreSQL | 建议本地或容器化部署 | 作为核心业务数据库 |
| Redis | 建议本地或容器化部署 | 作为缓存和任务辅助服务 |
| 迁移工具 | Alembic | 管理数据库结构演进 |
| 测试工具 | pytest | 用于基础测试和回归检查 |

### 3.3 容器化建议

若团队采用 Docker 方案，本地环境建议额外准备：

- Docker Desktop 或等价容器运行环境
- Docker Compose

适合容器化的基础服务：

- PostgreSQL
- Redis
- 未来的文件服务或辅助服务

## 4. 前端依赖管理建议

### 4.1 统一包管理器

前端依赖管理建议只保留一种主方案，不要在同一项目中混用多个包管理器。

建议策略：

- 团队统一一个包管理器
- 仓库中只保留一种锁文件
- CI 与本地开发使用同一安装命令

### 4.2 建议的前端依赖治理方式

- 使用明确的版本锁定策略
- 业务依赖与开发依赖分开管理
- 保持依赖最小化，不为“可能会用到”提前堆积包
- 组件库、状态管理、请求库、格式化和测试工具分层管理
- 版本升级走评审流程，避免自动升级导致构建失败

### 4.3 前端依赖分类建议

| 类别 | 建议 | 说明 |
| --- | --- | --- |
| 运行依赖 | Vue、Uni-app、Element Plus、Pinia 等 | 页面运行必须 |
| 请求依赖 | 统一请求封装所需库 | 用于调用后端 API |
| 工具依赖 | 日期、格式化、校验类工具 | 仅在确有需要时添加 |
| 开发依赖 | TypeScript、ESLint、Prettier、测试工具 | 仅用于开发与质量控制 |

### 4.4 前端版本控制建议

- 锁定 Node.js 大版本
- 锁定包管理器行为
- 锁定关键依赖的主版本
- 在锁文件变更时同步记录原因
- 前端构建失败时优先检查锁文件、Node 版本和包管理器版本

## 5. 后端依赖管理建议

### 5.1 统一 Python 依赖策略

后端建议建立明确的 Python 依赖管理策略，避免手工安装导致环境漂移。

推荐做法：

- 使用虚拟环境隔离项目依赖
- 将依赖声明文件纳入版本控制
- 锁定可复现版本
- CI 与本地开发使用同一套安装命令

### 5.2 后端依赖分类建议

| 类别 | 建议 | 说明 |
| --- | --- | --- |
| Web 框架 | FastAPI | 后端主框架 |
| 数据校验 | Pydantic | 请求、响应、结构化数据校验 |
| ORM / 数据访问 | SQLAlchemy 2.0 | 关系型数据库访问 |
| 迁移工具 | Alembic | 数据库结构版本管理 |
| 数据库驱动 | PostgreSQL 驱动 | 连接 PostgreSQL |
| 缓存 / 队列 | Redis 客户端、任务队列组件 | 连接 Redis、异步任务 |
| 测试 | pytest 等 | 基础测试与回归 |
| Excel 处理 | 解析库 | 仅在导入流程需要时加入 |

### 5.3 后端版本控制建议

- Python 版本必须明确
- 依赖文件必须可复现安装
- 重大依赖升级前需要兼容性评估
- 数据库驱动、ORM、迁移工具版本应协同升级
- 任务队列与 Redis 客户端版本应保持兼容

### 5.4 后端依赖约束

建议在后端依赖策略中明确：

- 业务代码不直接依赖本地全局 Python 环境
- 不在开发机上手工“试装”形成不可追踪状态
- 与 AI 或外部服务相关的 SDK 必须在有明确用途后再引入

## 6. 环境变量管理方案

### 6.1 管理目标

环境变量的核心目标是：

- 将配置与代码分离
- 将敏感信息与仓库分离
- 将不同环境的参数分离
- 方便本地、测试、生产切换

### 6.2 适合放入环境变量的内容

建议通过环境变量管理以下信息：

- 数据库连接地址
- Redis 连接地址
- JWT 或认证密钥
- API Key
- 外部模型服务地址与密钥
- 对象存储访问凭证
- 回调地址
- CORS 白名单
- 调试开关
- 日志级别
- 文件上传路径

### 6.3 不应提交到仓库的内容

以下内容必须避免进入代码仓库：

- 真实 API Key
- 数据库密码
- Redis 密码
- 对象存储 Secret
- 第三方平台密钥
- 私有回调签名密钥
- 生产环境连接串

### 6.4 推荐的环境文件分层

建议至少准备以下文件约定：

- `.env.example`：示例配置，只写字段名和默认占位值
- `.env.local`：本地开发专用，不提交仓库
- `.env.test`：测试环境配置，如需要可单独维护
- `.env.production`：生产环境配置，不随业务代码发布到公共仓库

### 6.5 环境变量字段建议

以下字段为基础建议，实际名称可按项目约定调整：

| 字段类别 | 建议字段 | 说明 |
| --- | --- | --- |
| 应用配置 | `APP_ENV`、`APP_NAME`、`DEBUG` | 控制运行模式 |
| 后端地址 | `API_BASE_URL` | 前端请求后端使用 |
| 数据库 | `DATABASE_URL`、`DB_HOST`、`DB_PORT`、`DB_NAME`、`DB_USER`、`DB_PASSWORD` | 数据库连接信息 |
| 缓存 | `REDIS_URL` | Redis 连接信息 |
| 安全 | `SECRET_KEY`、`JWT_SECRET` | 签名与加密相关 |
| 文件上传 | `UPLOAD_DIR`、`MAX_UPLOAD_SIZE` | 文件路径与限制 |
| 对象存储 | `OSS_ENDPOINT`、`OSS_BUCKET`、`OSS_ACCESS_KEY_ID`、`OSS_ACCESS_KEY_SECRET` | 存储服务配置 |
| AI / LLM | `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`、`LLM_TIMEOUT` | 模型服务配置 |
| 外部服务 | `DIFY_API_KEY`、`DIFY_BASE_URL`、`MCP_SERVER_URL` 等 | 外部能力接入配置 |

### 6.6 管理规范建议

- 真实密钥仅存放于本机、CI Secret 或部署平台 Secret
- `.env.example` 只做模板，不放真实值
- 环境变量必须有默认值或显式报错提示
- 读取不到关键变量时，应用启动应尽早失败并给出清晰提示
- 不同环境必须使用不同密钥，禁止复用生产密钥到本地

## 7. 本地开发前需要检查的工具

本地开发前，建议先确认以下工具可用。

### 7.1 通用工具

- Git
- 终端工具可用
- 网络访问正常
- 编辑器可用，并已安装必要插件

### 7.2 前端工具

- Node.js 版本正确
- 包管理器可用
- 浏览器可用
- 如需小程序调试，微信开发者工具可用
- 前端代码格式化、校验工具可用

### 7.3 后端工具

- Python 可用
- 虚拟环境可创建
- 数据库驱动安装正常
- PostgreSQL 可连接
- Redis 可连接
- pytest 可运行
- Alembic 可执行

### 7.4 容器工具

如采用 Docker：

- Docker Desktop 已启动
- Docker Compose 可用
- 本地端口未冲突
- 容器网络可正常访问数据库和 Redis

### 7.5 开发体验工具

建议检查：

- 代码格式化工具是否可执行
- 静态检查工具是否可执行
- Git 钩子是否存在冲突
- 本地 `.env` 文件是否已正确加载

## 8. AI / LLM / MCP / 外部服务接入前准备

### 8.1 AI / LLM 接入前准备

根据需求文档，AI 能力主要用于辅助 Excel 解析、字段识别、异常解释，不负责最终积分计算。

接入前建议准备：

- 可用的模型服务账号
- API Key 或访问令牌
- 基础模型名称与版本
- 调用地址与超时策略
- Token 消耗与预算预估
- 失败降级策略
- JSON 输出约束说明
- 结构化校验方案

### 8.2 MCP 接入前准备

如后续通过 MCP 对接外部能力，建议先确认：

- 目标 MCP 服务清单
- 每个服务的能力边界
- 认证方式
- 访问地址
- 超时与重试策略
- 是否允许本地开发环境访问
- 是否需要白名单或内网连接

### 8.3 外部服务接入前准备

如果项目需要接入对象存储、消息服务、第三方模型服务或其他平台，需先准备：

- 服务账号
- Key / Secret
- Bucket、空间名或项目名
- 回调地址
- IP 白名单要求
- 日志与审计要求
- 费用预算
- 故障降级预案

### 8.4 接入顺序建议

建议顺序如下：

1. 先完成本地环境与核心服务验证
2. 再接入数据库和 Redis
3. 再接入文件存储
4. 再接入 AI / LLM 服务
5. 最后接入 MCP 或其他外部编排能力

### 8.5 接入控制建议

- 先在测试环境验证
- 先做最小调用闭环
- 先验证鉴权，再验证功能
- 先验证超时、限流和错误返回，再进入业务集成
- AI 结果必须经过后端结构化校验，不可直接写库

## 9. 待确认项

以下内容在当前文档来源中尚未最终锁定，建议在立项或技术评审时确认：

1. 前端包管理器最终选择：`pnpm`、`npm` 或 `yarn`
2. 后端 Python 版本号
3. 后端依赖管理方案：`venv`、Poetry 还是 uv
4. 用户端最终是否同时包含 H5 与微信小程序
5. 是否强制使用 Docker 作为本地开发标准环境
6. 是否引入统一的对象存储服务，还是先使用本地文件系统
7. AI 模型服务最终选择：DeepSeek、OpenAI、其他模型平台或混合方案
8. MCP 是否为第一阶段必需能力
9. 测试环境与生产环境是否采用同一套部署方式
10. 是否需要统一的密钥管理平台

## 10. 本地最小验证清单

在开始业务开发前，建议至少通过以下验证。

### 10.1 基础环境验证

- [ ] Git 可用
- [ ] Node.js 可用
- [ ] Python 可用
- [ ] 包管理器可用
- [ ] Docker 可用，如项目要求容器化
- [ ] 浏览器可用
- [ ] 微信开发者工具可用，如需要小程序调试

### 10.2 服务连接验证

- [ ] PostgreSQL 可连接
- [ ] Redis 可连接
- [ ] 数据库可创建测试库
- [ ] 数据库迁移工具可运行
- [ ] 文件上传目录可写

### 10.3 配置验证

- [ ] `.env.example` 已存在
- [ ] 本地 `.env.local` 已创建但未提交
- [ ] 数据库连接信息已配置
- [ ] Redis 连接信息已配置
- [ ] API Key 已准备但未泄露到仓库
- [ ] 模型服务地址与密钥已配置，如需要

### 10.4 前端验证

- [ ] 前端依赖安装成功
- [ ] 前端开发服务可以启动
- [ ] 页面可以正常打开
- [ ] 可以完成一次基础 API 调用

### 10.5 后端验证

- [ ] 后端依赖安装成功
- [ ] 后端服务可以启动
- [ ] 健康检查接口可访问
- [ ] 日志输出正常
- [ ] 测试命令可运行

### 10.6 AI / 外部服务验证

如项目当前阶段需要接入：

- [ ] 模型服务可以完成一次最小请求
- [ ] JSON 返回格式可被后端解析
- [ ] 超时和错误处理可触发
- [ ] 凭证未被写入仓库

## 11. 建议的执行顺序

1. 确认技术选型和版本
2. 确认本地工具安装情况
3. 配置前端依赖管理
4. 配置后端依赖管理
5. 配置环境变量模板
6. 准备数据库和 Redis
7. 验证基础服务连接
8. 如有需要，再接入 AI / LLM / MCP / 外部服务

## 12. 结论

本项目的基础环境配置重点不在业务能力，而在“可重复、可验证、可隔离、可审计”。

建议在正式编码前，先统一以下内容：

- 运行时版本
- 依赖管理方式
- 环境变量规范
- 本地工具清单
- 外部服务接入边界

只有基础环境稳定后，再进入业务实现阶段，才能降低后续联调、部署和排障成本。
