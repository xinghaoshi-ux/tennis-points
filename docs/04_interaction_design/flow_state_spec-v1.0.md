# 流程状态规范 v1.0

> 本文定义 THA 年度赛事积分系统 MVP 阶段的关键状态清单、状态含义、迁移规则和可触发动作。
>
> 这些状态是前端页面设计、后端状态字段设计和 API 规范设计的共同依据。

---

## 1. 状态设计原则

- 每个业务对象有且仅有一个当前状态。
- 状态迁移必须通过明确动作触发，不允许隐式跳转。
- 状态名称在前端展示、后端字段和 API 响应中保持一致。
- 状态迁移不可逆时必须标注，可逆时必须说明回退条件。

---

## 2. 赛季状态（Season）

### 2.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 草稿 | draft | 赛季已创建但未激活，不参与积分计算 |
| 激活 | active | 当前有效赛季，所有积分归入此赛季 |
| 已关闭 | closed | 赛季结束，积分冻结，不再接受新数据 |

### 2.2 状态迁移

```text
draft ──[激活赛季]──→ active ──[关闭赛季]──→ closed
```

### 2.3 状态详细定义

#### draft（草稿）

| 维度 | 说明 |
|---|---|
| 含义 | 赛季已创建，信息可编辑，尚未生效 |
| 进入条件 | 管理员创建新赛季 |
| 退出条件 | 管理员执行"激活赛季"操作 |
| 可触发动作 | 编辑赛季信息、删除赛季、激活赛季 |
| 下一个可能状态 | active |

#### active（激活）

| 维度 | 说明 |
|---|---|
| 含义 | 当前有效赛季，系统内同一时间只能有一个 active 赛季 |
| 进入条件 | 管理员激活某个 draft 状态的赛季（同时将原 active 赛季关闭） |
| 退出条件 | 管理员执行"关闭赛季"操作，或激活新赛季时自动关闭 |
| 可触发动作 | 编辑赛季信息、关闭赛季、在此赛季下创建赛事和导入数据 |
| 下一个可能状态 | closed |

#### closed（已关闭）

| 维度 | 说明 |
|---|---|
| 含义 | 赛季已结束，积分数据冻结，仅供查看 |
| 进入条件 | 管理员关闭赛季，或新赛季激活时自动关闭 |
| 退出条件 | 不可逆（如需修改须重新激活，需管理员确认） |
| 可触发动作 | 查看赛季数据、导出排行榜 |
| 下一个可能状态 | 无（终态） |

### 2.4 约束规则

- 系统内同一时间最多一个 active 赛季。
- 激活新赛季时，原 active 赛季自动变为 closed。
- closed 赛季下不允许新增赛事、上传 Excel 或生成积分。

## 3. 赛事状态（Tournament）

### 3.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 草稿 | draft | 赛事已创建，信息可编辑，尚未录入结果 |
| 已完赛 | completed | 赛事结果已导入并确认 |
| 已发布 | published | 赛事积分已生成并计入排行榜 |

### 3.2 状态迁移

```text
draft ──[确认导入赛事结果]──→ completed ──[生成积分并发布]──→ published
```

### 3.3 状态详细定义

#### draft（草稿）

| 维度 | 说明 |
|---|---|
| 含义 | 赛事已创建，基础信息可编辑，尚无赛事结果 |
| 进入条件 | 管理员创建赛事 |
| 退出条件 | 该赛事关联的 Excel 导入被确认 |
| 可触发动作 | 编辑赛事信息、删除赛事、上传 Excel |
| 下一个可能状态 | completed |

#### completed（已完赛）

| 维度 | 说明 |
|---|---|
| 含义 | 赛事结果已录入，等待积分生成 |
| 进入条件 | 管理员确认导入该赛事的赛事结果 |
| 退出条件 | 系统生成积分记录并发布 |
| 可触发动作 | 查看赛事结果、修正数据、生成积分 |
| 下一个可能状态 | published |

#### published（已发布）

| 维度 | 说明 |
|---|---|
| 含义 | 赛事积分已生成并计入年度排行榜 |
| 进入条件 | 系统完成积分生成 |
| 退出条件 | 如需修正，管理员可撤回至 completed 状态 |
| 可触发动作 | 查看赛事结果和积分、撤回发布（需确认） |
| 下一个可能状态 | completed（撤回时） |

### 3.4 约束规则

- 只有 active 赛季下才能创建赛事。
- draft 状态的赛事可以删除，completed 和 published 状态不可删除。
- 撤回 published 赛事会同时删除该赛事关联的 entries_points 记录，需要重新刷新排行榜。

## 4. Excel 上传状态（Upload）

### 4.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 待处理 | pending | 文件已上传，等待解析 |
| 解析中 | parsing | 后台任务正在解析 Excel |
| 已解析 | parsed | 解析完成，等待管理员确认 |
| 解析失败 | parse_failed | 解析过程出错 |
| 已导入 | imported | 管理员确认后数据已入库 |
| 已取消 | cancelled | 管理员主动取消此次上传 |

### 4.2 状态迁移

```text
pending ──[触发解析]──→ parsing ──[解析成功]──→ parsed ──[确认导入]──→ imported
                            │                      │
                            ↓                      ↓
                       parse_failed            cancelled
                            │
                            ↓
                     [重新上传] → pending
```

### 4.3 状态详细定义

#### pending（待处理）

| 维度 | 说明 |
|---|---|
| 含义 | Excel 文件已上传到服务器，等待触发解析 |
| 进入条件 | 管理员上传 Excel 文件 |
| 退出条件 | 系统开始解析，或管理员取消 |
| 可触发动作 | 触发解析、取消上传、查看文件信息 |
| 下一个可能状态 | parsing、cancelled |

#### parsing（解析中）

| 维度 | 说明 |
|---|---|
| 含义 | 后台任务正在读取和校验 Excel 内容 |
| 进入条件 | 系统开始执行解析任务 |
| 退出条件 | 解析完成或解析出错 |
| 可触发动作 | 查看解析进度（前端轮询或 WebSocket） |
| 下一个可能状态 | parsed、parse_failed |

#### parsed（已解析）

| 维度 | 说明 |
|---|---|
| 含义 | 解析完成，结构化数据已生成，等待管理员审核确认 |
| 进入条件 | 解析任务成功完成 |
| 退出条件 | 管理员确认导入，或管理员取消 |
| 可触发动作 | 查看导入预览、修改选手匹配、忽略异常行、确认导入、取消 |
| 下一个可能状态 | imported、cancelled |

#### parse_failed（解析失败）

| 维度 | 说明 |
|---|---|
| 含义 | 解析过程中发生不可恢复的错误 |
| 进入条件 | 解析任务执行失败（文件损坏、格式严重不符等） |
| 退出条件 | 管理员查看错误信息后重新上传 |
| 可触发动作 | 查看错误日志、下载错误报告、重新上传新文件 |
| 下一个可能状态 | 无（终态，需重新上传产生新的 pending 记录） |

#### imported（已导入）

| 维度 | 说明 |
|---|---|
| 含义 | 数据已写入 event_results 表 |
| 进入条件 | 管理员确认导入 |
| 退出条件 | 无（终态） |
| 可触发动作 | 查看导入结果 |
| 下一个可能状态 | 无（终态） |

#### cancelled（已取消）

| 维度 | 说明 |
|---|---|
| 含义 | 管理员主动放弃此次上传 |
| 进入条件 | 管理员在 pending 或 parsed 状态下取消 |
| 退出条件 | 无（终态） |
| 可触发动作 | 查看取消记录 |
| 下一个可能状态 | 无（终态） |

### 4.4 约束规则

- 同一赛事不允许同时存在多个 parsing 状态的上传记录。
- parse_failed 不可重试，必须重新上传新文件。
- imported 后如需修正，应通过赛事结果修正流程处理，不通过上传状态回退。

## 5. 积分生成状态（Points Generation）

积分生成是系统内部流程，不对应独立的业务对象，但需要状态追踪。

### 5.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 待生成 | pending | 赛事结果已导入，等待积分生成 |
| 生成中 | processing | 系统正在计算积分 |
| 已完成 | completed | 积分记录已写入 entries_points |
| 生成失败 | failed | 积分生成过程出错 |

### 5.2 状态迁移

```text
pending ──[触发生成]──→ processing ──[生成成功]──→ completed
                             │
                             ↓
                          failed ──[重试]──→ processing
```

### 5.3 约束规则

- 积分生成依赖积分规则配置，如果缺少对应规则则生成失败。
- 生成失败可重试。
- 生成完成后，赛事状态从 completed 变为 published。

---

## 6. 排行榜状态（Ranking）

排行榜是衍生计算结果，状态较简单。

### 6.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 最新 | up_to_date | 排行榜数据与积分记录一致 |
| 待刷新 | stale | 有新积分数据尚未反映到排行榜 |
| 刷新中 | refreshing | 系统正在重新计算排行榜 |

### 6.2 状态迁移

```text
up_to_date ──[新积分写入]──→ stale ──[触发刷新]──→ refreshing ──[刷新完成]──→ up_to_date
```

### 6.3 约束规则

- 积分生成完成后，排行榜自动标记为 stale。
- 管理员可手动触发刷新，也可由系统在积分生成后自动刷新。
- 用户端始终展示最近一次刷新的排行榜数据。
- 刷新过程中用户端仍可访问旧数据，刷新完成后切换到新数据。

---

## 7. 管理员认证状态（Auth）

### 7.1 状态清单

| 状态 | 英文标识 | 含义 |
|---|---|---|
| 未登录 | unauthenticated | 未持有有效凭证 |
| 已登录 | authenticated | 持有有效 Token |
| Token 过期 | expired | Token 已失效，需重新登录 |

### 7.2 状态迁移

```text
unauthenticated ──[登录成功]──→ authenticated ──[Token 过期]──→ expired
                                      │                            │
                                      ↓                            ↓
                               [主动登出] → unauthenticated   [重新登录] → authenticated
```

## 8. 状态与系统设计的映射关系

### 8.1 状态 → 后端字段

| 业务对象 | 数据库字段 | 字段类型 | 可选值 |
|---|---|---|---|
| Season | status | VARCHAR | draft / active / closed |
| Tournament | status | VARCHAR | draft / completed / published |
| Upload | status | VARCHAR | pending / parsing / parsed / parse_failed / imported / cancelled |

### 8.2 状态 → 前端页面

| 状态 | 前端展示 |
|---|---|
| Season.active | 顶部栏显示当前赛季名称 |
| Tournament.draft | 赛事列表中标记"草稿"标签 |
| Tournament.published | 赛事列表中标记"已发布"标签 |
| Upload.parsing | 步骤条高亮"解析"步骤，显示加载动画 |
| Upload.parsed | 步骤条高亮"预览"步骤，展示预览表格 |
| Upload.parse_failed | 步骤条"解析"步骤标红，展示错误信息 |
| Upload.imported | 步骤条全部完成，显示成功提示 |
| Ranking.stale | 排行榜管理页显示"有待刷新数据"提示 |

### 8.3 状态 → API 响应

API 响应中的 status 字段必须使用本文档定义的英文标识，前端根据 status 值决定页面展示逻辑。

示例：

```text
GET /api/v1/admin/uploads/{id}
响应：{ "id": "...", "status": "parsed", ... }

前端根据 status === "parsed" 展示导入预览界面。
```

## 9. 人工介入节点

以下状态迁移需要管理员人工确认：

| 迁移 | 人工操作 | 操作后状态变化 |
|---|---|---|
| parsed → imported | 管理员确认导入 | Upload 变为 imported，触发积分生成 |
| completed → published | 管理员确认发布（或系统自动） | Tournament 变为 published |
| active → closed | 管理员关闭赛季 | Season 变为 closed |
| published → completed | 管理员撤回发布 | Tournament 回退，关联积分记录删除 |

---

## 10. 状态一致性保障

- 上传确认导入后，系统应在同一事务中完成：Upload 状态变更 + event_results 写入。
- 积分生成完成后，系统应在同一事务中完成：entries_points 写入 + Tournament 状态变更。
- 排行榜刷新应在计算完成后原子切换，避免用户看到中间状态。
